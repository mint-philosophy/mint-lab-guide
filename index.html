🏠 What Is This System?

The MINT Lab workspace is a research automation platform built around Minty, an AI research management agent. It runs on a Mac Studio and orchestrates 13 background daemons that automate tasks which would otherwise consume hours of researcher time — fetching, classifying, ingesting, indexing, summarizing, and distributing academic content — while enabling entirely new capabilities like deep multi-agent corpus analysis that simply weren't possible before. Research superpowers, not a replacement for researchers.

Core Idea

Information flows through a pipeline: external sources (Twitter, arXiv, RSS, Bluesky, email, LinkedIn) are automatically fetched, classified by relevance using Claude Opus, and posted to Slack. Papers are downloaded, converted to markdown via a 6-tier cascade, embedded in a vector database, analyzed across 41 semantic dimensions, and made searchable via a Slack bot. A daily digest email goes out to 25+ lab members.

Key Design Principles

  • Delegation-first: The main agent orchestrates; subagents do the work
  • Slack as message bus: All daemons communicate through Slack channels
  • CLI subscriptions, not API: All LLM calls go through claude -p or Codex CLI
  • Graceful degradation: Every stage can fail without stopping the pipeline
  • Single-tier vault: All credentials in macOS login keychain, direct access for all agents and daemons

⚡ The Content Pipeline

Here's how information flows from external sources to lab members. Slack's #news channel is the central hub — four daemons write to it, three daemons read from it.

Key insight: The system forms a funnel. External sources produce hundreds of items daily. feedme-daily classifies them down to ~20-40%. Seth's thumbs-up on #feed curates further. Everything that reaches #news gets PDFs attached, ingested into the corpus, and summarized in the daily digest.

⚙ The 13 Daemons

All daemons run on the Mac Studio via macOS launchd. They share a common platform_config.py for paths and secrets, and a shared Python 3.12 venv at daemons/pipeline/.venv/.

slack-index / search-minty
4h index Persistent bot ●

Two-part system. The indexer runs every 4 hours, incrementally embedding all public Slack messages into ChromaDB via Voyage AI voyage-3, with per-channel watermarks and thread replies concatenated into parent documents.

The search bot (@search-minty) is a persistent Socket Mode daemon. When @mentioned, it spawns a full Claude Opus agent with Bash-only tool access. The agent classifies intent, generates 3–5 semantically diverse query variants, executes them via search_dispatch.py, and synthesizes results into flowing prose with inline Slack permalink citations.

Search strategyMulti-query Reciprocal Rank Fusion (RRF, k=60). Agent generates diverse-angle queries, not synonyms. Scores fused across retrieval passes.
Resilience3-tier: Primary agent → Relief officer (fresh context) → Direct search + formatter. Never returns a hard error.
UXHaiku-generated contextual placeholder while Opus works. Whimsical progress messages after 2 min. Catch-up sweep on restart recovers missed queries.
Filtersin:#channel, from:user, days:N. Self-referential detection: "my posts" auto-maps to requesting user.
IndexChromaDB at data/slack_index/, Voyage voyage-3 embeddings
Key filesindex_slack.py, watcher.py, search_dispatch.py, agents/search_agent.md

RAPTOR Summaries

Hierarchical summaries per paper: a document-level overview (~300 words) plus section summaries for Introduction, Methods, Results, Discussion, and Conclusion. These serve dual purposes: fast search index (via document_summary_embedding) and context-efficient paper reading (agents read summaries instead of full text).

Deep Research Pipeline

The CO follows a recommended playbook but can deviate — skipping phases for narrow queries or adding search rounds for broad ones. The standard pipeline:

PhaseWorkersWhat happens
1. SearchSearch footsoldierIterative corpus search (min 3 rounds, self-assessment). CO launches retrieve + search lieutenant in parallel.
2. RetrieveRetrieve footsoldierFetches full texts for all found papers. Runs parallel with search lieutenant assessment.
3. LandscapeLandscape + Landscape AnalysisAssembles 41Q landscape data from corpus (no LLM), then LLM-driven field mapping and thematic analysis.
4. GroupingGrouping footsoldierStrategic thematic clustering into groups for parallel reading. Produces Round 1 (thematic) and Round 2 (recombined) group assignments.
5. Reading R1Up to 5 parallel readersEach reader analyzes papers in its thematic group. Reports include cross-paper tensions, shared arguments, and gaps.
6. Reading R2Up to 5 parallel readersPapers recombined into cross-cutting groups. R2 readers receive R1 reports to challenge and extend.
7. AssessmentReading lieutenantAssesses per-reader quality, flags thin reports, identifies key themes for synthesis.
8. SynthesisSynthesis footsoldier + lieutenantUnified thematic synthesis of all R1 and R2 reports. Lieutenant does final quality check.
9. Peer ReviewPeer review footsoldier (optional)Dual review: intellectual critique + fact-checking. Can be skipped for speed.
10. Citation ReviewCitation review footsoldierVerifies all paper citations are Drive links, checks for hallucinated references.

Resilience: If the CO exhausts its context window, a relief officer is spawned with the CO's progress. If that also fails, a legacy fallback runs the pipeline as a monolithic script. The system never returns empty-handed if papers were found.

Delegation Protocol

Mandatory rule: "Every tool call you make is context you burn. Subagents have FRESH context. If a task can be described in a prompt, delegate it." The only valid reason to do work yourself is "this literally cannot be described in a prompt."

  • Web search, documentation lookup, file processing, batch operations → always delegate
  • Model policy: Opus for substantive work, Sonnet for retrieval-only, Haiku only for startup data gathering
  • Hooks enforce this: delegation-nudge.py warns after 15 consecutive Read/Grep/Glob calls without a Task delegation

The Iron Laws

Foundational rules that override all other behavior:

Law 0: No Shortcuts
When asked for "all" — do ALL. Cherry-picking = failure.
Law 1: Evidence Over Claims
"It should work" = failure. Verify now, not later.
Law 2: Document Everything
If not written down, didn't happen.
Law 3: Persistence
Try 3+ approaches before asking. Solve, don't report.
Law 5: Complete the Checklist
"Partially done" = not done.
Law 6: Think Before Acting
Never rm — use trash. Map dependencies first.
